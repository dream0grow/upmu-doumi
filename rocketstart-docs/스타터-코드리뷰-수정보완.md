# 로켓스타트 웹 MVP 스타터 — 코드 리뷰 & 수정·보완 목록

> 업로드하신 `rocketstart-web-mvp` 스타터를 전부 읽고, **무엇을 고치고 무엇을 더해야
> MVP가 실제로 돌아가는지**를 우선순위로 정리했습니다.
> (제품 정의 출처: 스타터 안의 `IMPLEMENTATION_BRIEF.md`)

---

## 0. 먼저 — 이 스타터는 잘 만들어졌습니다

칭찬부터 하면(그대로 살릴 부분):

- **데이터 모델이 탄탄합니다.** `supabase/schema.sql`의 Project→RocketTask→Deliverable→Subtask→FocusSession
  관계, RLS 정책, `rocket_task_metrics` 뷰까지 이미 설계가 좋습니다.
- **핵심 계산 로직이 분리돼 있습니다.** `lib/rocket/metrics.ts`(90% 산출물, 뽀모도로 예측정확도)에
  단위 테스트까지 있습니다.
- **화면 5개 뼈대**(오늘/로켓 만들기/타이머/리포트/로그인)와 **PWA 스캐폴드**(manifest, sw.js),
  **Web Audio 백색소음**까지 들어 있습니다.

→ 지금 필요한 건 "새로 짜기"가 아니라 **빈칸 채우기**입니다. 아래 순서대로 하면 됩니다.

## 2. 🟠 P1 — 브리프에 있는데 아직 없는 "핵심 MVP 기능"

스타터의 `IMPLEMENTATION_BRIEF.md`가 스스로 요구하는데 **아직 구현 안 된 것들**입니다.
이게 사실상 진짜 "수정 보완 목록"의 핵심입니다.

### P1-1. 복구 모드 화면이 통째로 없음 ⭐가장 중요
- 브리프 "화면 우선순위 4번 = 복구 모드"인데 **해당 페이지가 없습니다.**
  (`today`는 '복구 필요' 카드만 보여줄 뿐, 복구 흐름이 없음)
- 이건 이 앱의 **차별점**입니다. "미루면 방치 대신 복구 질문."
- **추가할 것**: `/recovery` 화면
  - 막힘 이유 선택(스키마 `blocker_reason` enum 그대로: 미착수/과업 과다/자료 부족/에너지 부족/두려움/완벽주의/일정 충돌…)
  - "다음 15분 최소 행동" 입력
  - 저장 → `reflection_logs` 테이블

### P1-2. 성찰·미룸 메모 기록 UI 없음
- 스키마 `reflection_logs`는 있는데 **입력 화면이 없습니다.**
- 타이머 종료·미착수·하루 끝에 "왜 미뤘나 / 다음 최소 행동" 한 줄을 남기는 흐름 필요.

### P1-3. 하위 과업(Subtask) 입력이 폼에 없음 — 데이터 모델과 충돌
- 브리프: "산출물별 **하위 과업** 생성 → **하위 과업별 예상 뽀모도로**".
- 스키마: `estimated_pomodoros`는 **subtasks 테이블**에 있음.
- 그런데 `new-rocket/page.tsx`는 **산출물(deliverable)에 직접** 예상 뽀모도로를 입력받음.
- **결과**: 지금 폼 구조대로 저장하면 스키마와 안 맞습니다.
- **해결**: 폼을 `산출물 → 그 안에 하위 과업 목록 → 하위 과업에 예상 뽀모도로` 2단 구조로.

### P1-4. "90% 완료 확정" 버튼 없음
- 브리프: "사용자가 마지막에 90% 완료 확정 버튼을 누른다" (`rocket_tasks.confirmed_90_at`, `is_90_done`).
- 어느 화면에도 확정 버튼이 없습니다. 오늘/리포트 카드에 추가 필요.

### P1-5. 저장(CRUD)이 전부 미연결 — 앱이 아직 "읽기 전용 목업"
- `new-rocket`의 저장 버튼: "저장 연결은 다음 마일스톤" (동작 안 함).
- `today`·`report`는 `mock-data.ts`만 표시.
- 타이머 "세션 기록": `alert`만 뜨고 저장 안 됨.
- **이게 정상입니다**(브리프의 "2번째 마일스톤"). 다만 **MVP가 되려면 최소 한 바퀴는 저장이 돌아야** 함:
  로켓 생성 → 저장 → 오늘 화면에 실제로 뜸 → 타이머 세션 저장 → 리포트에 반영.

### P1-6. 타이머가 특정 로켓/하위 과업과 연결 안 됨
- `focus_sessions.task_id`는 **NOT NULL**인데, 타이머 화면엔 "어느 로켓을 집중하는지" 고르는 UI가 없음.
- **추가**: 타이머 진입 시 대상 로켓(또는 하위 과업) 선택 → 세션 저장 시 그 id 사용.

### P1-7. 로그인·인증 가드 없음
- `login/page.tsx`는 **화면만** 있고 Supabase Auth 미연결.
- `(dashboard)` 그룹에 **인증 가드(middleware)가 없어** 로그인 없이도 다 열림.
- **추가**: `middleware.ts`로 세션 확인 + 보호 라우트, 로그인 실제 연결(이메일/Google).

## 3. 🟡 P2 — 논리·정합성 버그 (지금 고쳐두면 나중에 안 터짐)

### P2-1. 타입(TS)과 DB(enum)가 안 맞음 ⭐
- `lib/rocket/types.ts`:
  - `SprintDayPolicy = 'include' | 'exclude'` → 스키마 enum엔 **`'ask_each_rocket'`도 있음**(그리고 `user_settings` 기본값이 그것!).
  - `RocketStage`가 **한글 문자열**(`'1일차 구조 잡기'`…) → 스키마는 **영문 enum**(`'day1_structure'`…).
- **결과**: Supabase 붙이는 순간 값이 안 맞아 저장/조회가 깨집니다.
- **해결**: DB enum을 정본으로 두고, **화면 표시용 한글은 매핑 함수**로 분리
  (`stageLabel(stage): string`). `new-rocket` 폼의 3일 계산 방식 select에도 `ask_each_rocket` 옵션 추가.

### P2-2. 뽀모도로 라벨 '데이터 부족'인데 사용률·정확도가 0으로 표시됨
- `metrics.ts`: 예상>0, 실제=0이면 `label='데이터 부족'`이지만 `usageRatio=0`, `accuracyScore=0`을 반환.
- 그래서 리포트에 **"판단: 데이터 부족 / 사용률 0% / 예측 정확도 0점"** 처럼 모순돼 보임(mock task-3).
- **해결**: 실제=0일 때 `usageRatio`·`accuracyScore`를 **`null`로** 반환해 "데이터 없음"으로 일관 표시.

### P2-3. `new-rocket` 상태를 직접 변형(mutate)
- `const next = [...deliverables]; next[index].title = ...` → 배열은 복사했지만 **안의 객체는 그대로**라 직접 수정.
- 지금은 동작하지만 React 안티패턴. 항목 객체까지 새로 만들어 교체하는 게 안전.
- 숫자 입력도 빈 값일 때 처리(예: `Number(v) || 0`) 넣기.

### P2-4. 리포트가 로켓별만, 프로젝트별 집계 없음
- 브리프: "**프로젝트별**, 로켓별" 리포트.
- 지금은 로켓별만. `metrics.ts`에 `aggregatePomodoroEstimates`가 있는데 **아무 데서도 안 씀** → 프로젝트 집계에 활용.

| P3-2 | **서비스워커 미등록** | `sw.js`는 있는데 등록 코드가 없음 → 푸시/오프라인 동작 안 함. 클라이언트에서 `navigator.serviceWorker.register('/sw.js')` |

| P3-7 | **캘린더/푸시 env만 있고 미구현** | 브리프대로 **후반 마일스톤**. MVP에선 비워둬도 됨(우선순위 낮음) |

## 5. 추천 작업 순서 (브리프 마일스톤에 맞춤)

브리프의 "1·2·3 마일스톤"에 위 항목을 끼워 넣은 실전 순서입니다.

**1주차 — "빌드되고, 한 바퀴가 돈다"**
1. `[P0-1~3]` 의존성 고정 + eslint + .gitignore → `npm run dev/lint/typecheck/test` 그린 만들기
2. `[P2-1]` 타입↔DB enum 정합 + stage 한글 매핑 함수
3. `[P2-2]` metrics 데이터부족 시 null 처리 + 테스트 보강
4. `[P1-3]` new-rocket 폼을 산출물→하위과업 2단 구조로
5. `[P1-1]` **복구 모드 `/recovery` 화면** 추가(이게 차별점)

**2주차 — "실제로 저장된다" (Supabase 연결)**
6. `[P1-7]` 로그인 연결 + `middleware.ts` 인증 가드
7. `[P1-5]` 로켓 생성 저장 → `today`를 실데이터로
8. `[P1-6]` 타이머를 로켓과 연결 + `focus_sessions` 저장
9. `[P1-2]` 성찰/미룸 메모 저장(`reflection_logs`)
10. `[P1-4]` 90% 완료 확정 버튼 + 리포트 실데이터화 `[P2-4]`

**3주차 — "앱처럼 보인다"**
11. `[P3-1~3]` PWA 아이콘·SW 등록·타이머 알림
12. `[P3-4,6]` 볼륨·설정 화면
13. 모바일 점검 + 지인 5명 테스트

## 6. 한 줄 결론

> 스타터의 **설계는 이미 훌륭합니다.** 지금 필요한 건
> ① 빌드 안정화(P0) → ② 브리프가 요구하는 **복구 화면·하위과업·저장 연결**(P1) → ③ 타입/DB 정합(P2).
> 특히 **복구 모드 화면(P1-1)** 이 이 앱을 "또 하나의 할일 앱"과 다르게 만드는 핵심이니
> MVP 안에 꼭 넣으시길 권합니다.
