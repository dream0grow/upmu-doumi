# 개발 로드맵 — 로켓스타트 3주 계획

> 이미 만들어진 `rocketstart-web-mvp` 스타터(Next.js App Router + TypeScript + Supabase, 화면 5개 뼈대·`supabase/schema.sql`·`lib/rocket/metrics.ts`·백색소음 컴포넌트 포함)를 기준으로 짠 3주 로드맵입니다. 퇴근 후 개발(평일 1~2시간, 주말에 몰아서)을 현실적으로 반영했습니다. 매주 끝에 "이번 주 완료 기준"을 두어 다음 주로 미루기 쉬운 항목을 걸러냅니다.

## 사용법

- Must = 이 주에 반드시 끝내야 다음 주가 가능함
- Should = 되면 좋음, 안 되면 다음 주 초로 이월
- Could = 여유 있으면, 없으면 백로그로 보류
- 체크박스는 그대로 두고 커밋 메시지나 이슈 제목에 복사해서 써도 됩니다.

---

## 1주차 — 빌드 안정화 + 화면 mock 완성

**목표:** 로그인 없이도, 실제 데이터 없이도 5개 화면이 mock 데이터로 끊김 없이 동작한다. "이 앱이 뭘 하는 앱인지" 클릭만으로 보여줄 수 있는 상태.

### Must
- [ ] `rocketstart-web-mvp`가 로컬에서 `npm run dev`로 뜨는지 확인 (환경변수 없이도 mock 화면은 떠야 함)
- [ ] 오늘(`/today`) 화면: 오늘 착수 / 오늘 90% 마감 / 복구 필요 / 진행 중 4버킷이 `mock-data.ts` 기준으로 올바르게 분리되는지 확인
- [ ] 로켓 만들기(`/new-rocket`) 화면: 시작일·계산 방식 바꿨을 때 90% 목표일·총 예상 뽀모도로가 `lib/rocket/dates.ts` 계산으로 자동 갱신되는지 확인
- [ ] 집중 타이머(`/timer`) 화면: 15/25/50분 버튼 + 사용자 지정 입력이 실제로 카운트다운되는지, 완료 시 알림이 뜨는지 확인
- [ ] 백색소음 5종(화이트/브라운/빗소리/카페/무음) 재생·정지 동작 확인 (`WhiteNoisePlayer.tsx`)
- [ ] 리포트(`/report`) 화면: `metrics.ts`의 판단 라벨(정확함/과소예측/과대예측/데이터 부족) 4가지 케이스를 mock 데이터로 최소 1개씩 재현해서 눈으로 확인
- [ ] `lib/rocket/metrics.test.ts` 통과 확인 (`npm test`)

### Should
- [ ] 복구 모드(`/recovery`) 화면 뼈대 신규 작성 (스타터에는 아직 없음) — 막힘 이유 9종 버튼 + 다음 15분 최소 행동 입력 UI만, 저장은 2주차에 연결
- [ ] 오늘 화면에 빈 상태("아직 로켓이 없어요") 컴포넌트 추가
- [ ] 모바일 폭(375px)에서 카드 레이아웃 깨짐 없는지 확인

### Could
- [ ] 상단 내비에 현재 페이지 강조(active 스타일) 추가
- [ ] 로켓 카드에 스테이지별 색상 뱃지 아이디어만 스케치

**이번 주 완료 기준:** 배포 없이 로컬에서 5개 화면을 순서대로 눌러보면서 "이게 로켓스타트다"라고 설명할 수 있으면 통과.

## 2주차 — Supabase CRUD 연결

**목표:** mock 데이터를 실제 Supabase 테이블 read/write로 교체한다. 로그인은 아직 없어도 되고, 개발자 본인 계정 하나로만 동작해도 됨.

### Must
- [ ] Supabase 프로젝트 생성 + `supabase/schema.sql` 그대로 적용 (타입·테이블·RLS 정책 포함)
- [ ] `.env.example` 참고해서 로컬 `.env.local` 구성, `lib/supabase/client.ts`·`server.ts` 연결 확인
- [ ] 로켓 만들기 폼 제출 → `projects`, `rocket_tasks`, `deliverables`, `subtasks` 테이블에 실제 저장
- [ ] 오늘 화면을 mock 대신 실제 `rocket_tasks` + `deliverables` 조회로 전환, `todayBucket` 분류 로직을 서버 쪽 계산으로 이동
- [ ] 집중 타이머 세션 종료 시 `focus_sessions`에 `completed_pomodoros`, `actual_minutes`, `sound_type` 저장
- [ ] 리포트 화면을 `rocket_task_metrics` 뷰(스키마에 이미 정의됨) 기반으로 전환
- [ ] "90% 완료 확정" 버튼 → `rocket_tasks.is_90_done = true`, `confirmed_90_at` 기록

### Should
- [ ] 복구 모드 저장 → `reflection_logs`에 `blocker`, `next_minimum_action`, `trigger` 저장 연결
- [ ] 산출물 체크(done/todo/doing/skipped) 상태 변경 시 실시간으로 진행바 갱신
- [ ] 하위과업 완료 체크 UI 추가 (스타터에는 목록만 있고 체크 UI 없음)

### Could
- [ ] RLS 정책 실제로 다른 테스트 계정으로 침투 테스트(내 데이터가 안 보이는지 확인)
- [ ] `time_blocks`, `reminder_events` 테이블 스키마만 미리 검토 (3주차 대비)

**이번 주 완료 기준:** 브라우저를 새로고침해도 로켓·산출물·집중 세션 기록이 그대로 남아있으면 통과. mock 데이터 파일은 삭제하지 않아도 되지만 화면에서는 더 이상 사용하지 않음.

## 3주차 — 로그인 · PWA · 복구/성찰 마무리

**목표:** 나 혼자만 쓰는 게 아니라 "계정으로 로그인해서 쓰는 앱"이 되고, 홈 화면에 설치 가능한 PWA가 되며, 복구 흐름이 완결된다.

### Must
- [ ] Supabase Auth 이메일 로그인 연결 (`/login` 화면 실제 동작)
- [ ] 로그인 안 한 사용자는 `/today` 등 대시보드 접근 시 `/login`으로 리다이렉트
- [ ] 로그아웃 버튼 동작 확인
- [ ] 복구 모드 전체 흐름 완결: 막힘 이유 선택 → 최소 행동 작성 → 저장 → "15분 타이머 바로 시작" 버튼이 실제로 `/timer`로 값 전달
- [ ] PWA manifest 작성(`public/manifest.json`) + 아이콘 + `next-pwa` 또는 자체 service worker 최소 캐싱 적용
- [ ] 모바일 브라우저에서 "홈 화면에 추가"로 설치되는지 실기기 확인

### Should
- [ ] Google 로그인 연결
- [ ] 오늘 화면 진입 시 "복구 필요" 로켓이 있으면 부드러운 알림 배지 표시
- [ ] 브라우저 알림 권한 요청 + 미착수 로켓에 대한 최소 리마인더 1종

### Could
- [ ] Google Calendar 연동 초안 (`time_blocks.provider = 'google'` 저장까지만)
- [ ] Apple 로그인 설정 준비(실제 연동은 보류, 버튼만 비활성 표시)
- [ ] 알림 강도 설정(`user_settings.notification_intensity`) UI 연결

**이번 주 완료 기준:** 새 계정으로 회원가입 → 로켓 만들기 → 타이머 세션 1개 → 리포트 확인 → 복구 모드까지, 로그인부터 끝까지 한 번에 끊김 없이 돌면 MVP 통과.

## 혼자 개발(퇴근 후) 팁

- 평일에는 Must 체크박스 1~2개만 목표로 잡기. 하루 몰아서 다 하려고 하면 로켓스타트 자체가 말하는 "미룸"에 빠지기 쉽습니다.
- 주말에 그 주 Should 항목을 몰아서 처리하는 흐름이 현실적입니다.
- 매주 일요일 저녁, 이 로드맵 문서를 열어 체크된 것과 안 된 것을 눈으로 확인하고 다음 주 Must를 다시 정하세요. (이 습관 자체가 로켓스타트 리포트 화면이 하려는 일과 같습니다.)
