# RocketStart Web MVP — 제품화 비판적 리뷰

> 대상: `dream0grow/rocketstart-web` (Manus로 생성한 실행관리 웹앱 MVP)
> 관점: 코드 결함 + 실전 제품화 관점에서 비판적으로.
> 작성일: 2026-07-01

## 0. 한 줄 요약

**화면과 디자인은 완성도가 높지만, "산출물을 완료 처리해서 90%에 도달한다"는 이 제품의 핵심 루프가 UI에 존재하지 않는다.** 즉 사용자는 로켓을 만들 수만 있고, 진행/완료시킬 수 없다. 백엔드 API(`confirm90`, `setDeliverableStatus`, `updateStage`, `remove`)는 다 만들어 놓고 프런트에서 아무 데서도 호출하지 않는다. 여기에 집중 타이머의 세션 기록 버그(정상 완료 시 기록 누락)까지 겹쳐, 데모는 되지만 실사용은 되지 않는 상태다. 아래에 심각도 순으로 정리한다.

---

## 1. 심각 — 핵심 기능이 동작하지 않음 (제품 성립 불가)

### 1-1. 산출물 완료·90% 확정·단계 변경 UI가 통째로 없음 ★최우선

- 서버에는 프로시저가 다 있다: `server/routers.ts:176` `setDeliverableStatus`, `:165` `confirm90`, `:142` `updateStage`, `:192` `remove`.
- 그런데 클라이언트 전체에서 실제로 호출되는 mutation은 `rocket.create`, `focus.recordSession`, `reflection.create` 셋뿐이다(검색으로 확인).
- 결과:
  - 산출물 상태를 `done`으로 바꿀 방법이 없다 → `calculateRequiredDeliverableProgress`(`shared/rocket.ts:88`)의 `completedWeight`가 **항상 0** → **오늘 화면·리포트의 달성률 막대가 영구히 0%**.
  - `isDone90`을 확정할 방법이 없다 → 90% 목표일이 지나도 로켓은 계속 `progress`/`due90` 버킷에 남는다. `classifyTodayBucket`(`shared/rocket.ts:218`)의 `done` 처리 분기가 도달 불가.
  - 로켓 단계 진행(1일차→2일차→90%)도, 로켓 삭제도 UI에 없다.
- 즉 "로켓 만들기" 이후의 **모든 상태 전이가 막혀 있다.** 리포트의 숫자는 예상 뽀모도로/실제 뽀모도로 외에는 전부 0으로 고정된다.
- **필요 작업**: 로켓 상세/카드에 산출물 체크박스(→`setDeliverableStatus`), "90% 확정" 버튼(→`confirm90`), 단계 이동, 삭제 버튼을 연결. 이게 없으면 나머지 리뷰는 의미가 약하다.

### 1-2. 집중 타이머 — 정상 완료된 세션이 기록되지 않음 (stale closure)

`client/src/pages/Timer.tsx:115` 의 `setInterval` 콜백은 `toggleRun`이 실행된 시점 렌더의 `finishSession`을 캡처한다. 그 `finishSession`(`:72`)이 참조하는 `remaining`도 그 시점 값(= `plannedMinutes*60`, 가득 찬 값)으로 **고정**된다.

```
const elapsedSec = plannedMinutes * 60 - remaining; // remaining이 stale(가득참) → 0
const actualMinutes = Math.round(elapsedSec / 60);   // 0
if (actualMinutes >= 1) { ... recordMutation.mutate ... } // 통과 못 함
```

- 타이머가 **자연 종료(가장 흔한 성공 경로)** 되면 `elapsedSec = 0` → `actualMinutes = 0` → 기록이 저장되지 않는다. 알람만 울리고 세션은 사라진다.
- 아이러니하게도 사용자가 중간에 "세션 기록"을 누르면(`:258`, 최신 렌더의 `finishSession(false)`) 그 경로는 정상 동작한다.
- 이 제품은 "실제 사용 시간을 예측 정확도 데이터로 학습"하는 게 리포트의 핵심인데, 그 데이터의 대부분(완주한 뽀모도로)이 유실된다.
- **수정 방향**: `remaining`을 클로저로 읽지 말고 `finishSession`을 `useRef`로 최신화하거나, `setRemaining` 업데이터 안에서 경과 시간을 계산해 넘기거나, 경과 시간을 별도 `useRef`/타임스탬프(`startedAt`)로 관리.

### 1-3. 타이머 로직 전반이 setState 클로저에 취약

- `finishSession`이 `setRemaining` 업데이터(`:117`) **내부에서** 다시 호출되어 `setRunning`/`clearInterval`/`setRemaining`을 연쇄로 부른다. 렌더 도중 다른 setState를 부르는 구조라 React 경고/이중 실행 위험이 있다.
- 프리셋 변경 `useEffect`(`:61`)와 타이머 실행 상태가 얽혀, 실행 중 언마운트/재마운트 시 경과시간 계산이 어긋날 수 있다.
- 전반적으로 타이머는 "경과 시간(초)"을 단일 소스(예: `startTimestamp`)로 두고 파생값을 계산하는 구조로 리팩터링하는 편이 안전하다.

## 2. 데이터·백엔드 결함

### 2-1. 생성이 트랜잭션이 아님 (부분 저장 위험)

`rocket.create`(`server/routers.ts:85`)는 `createProject` → `createRocketTask` → 산출물 루프 → `createSubtasks`를 **개별 쿼리**로 순차 실행한다. 중간에 실패하면 프로젝트만, 혹은 로켓만 남는 **고아 레코드**가 생긴다. MySQL 트랜잭션(`db.transaction`)으로 묶어야 한다.

### 2-2. 삭제도 트랜잭션이 아니고, 프로젝트는 영원히 안 지워짐

- `deleteRocketTask`(`server/db.ts:175`)는 reflection→session→subtask→deliverable→task를 5개의 개별 DELETE로 지운다. 트랜잭션이 없어 중간 실패 시 고아가 남는다.
- 더 큰 문제: `create`는 로켓마다 **매번 새 `projects` row를 만든다**(`routers.ts:94`). 그런데 삭제 시 `projects`는 건드리지 않고, `overview`도 `projects`를 읽지 않는다. 결국 `projects` 테이블은 **쓰기 전용 쓰레기**로 무한히 쌓인다. 프로젝트 개념을 실제로 쓸 거면 재사용/선택 UI가 필요하고, 안 쓸 거면 스키마에서 빼야 한다.

### 2-3. 외래키(FK)·인덱스 전무

마이그레이션(`drizzle/0001_*.sql`)에 **FK 제약도, 보조 인덱스도 하나도 없다.**
- 모든 조회가 `WHERE userId = ?` / `taskId IN (...)` 인데 `userId`, `taskId`, `deliverableId` 어디에도 인덱스가 없어 **전부 풀 테이블 스캔**. 사용자가 늘면 `overview` 한 번에 4개 테이블을 스캔한다.
- FK가 없어 앱 코드가 수동으로 정합성을 지켜야 하고(2-2), 그 코드가 트랜잭션도 아니다.
- **필요**: `(userId)`, `(taskId)`, `(userId, taskId)` 복합 인덱스 추가, 가능하면 `ON DELETE CASCADE` FK.

### 2-4. taskId 소유권 미검증 (데이터 무결성)

- `focus.recordSession`(`routers.ts:205`)과 `reflection.create`(`:236`)는 `taskId`를 받아 그대로 저장하지만, **그 taskId가 요청 사용자 소유인지 검증하지 않는다.** 존재하지 않거나 남의 taskId를 참조하는 row를 만들 수 있다.
- 읽기 경로가 `userId`로 스코프되어 있어 정보 유출로 이어지진 않지만, 쓰레기/불일치 데이터가 쌓인다. 삽입 전 `getRocketTask(userId, taskId)`로 확인하는 게 맞다.

### 2-5. 입력 크기 상한 없음 (경미한 DoS)

`createRocketInput`(`routers.ts:23`)의 `deliverables`는 `min(1)`만 있고 상한이 없다. 산출물마다 개별 INSERT를 도는 구조(`:115`)라, 수천 개를 담아 보내면 그만큼 쿼리가 돈다. `outcome90` 등 텍스트도 상한이 없다. `.max()` 상한을 넣는 게 좋다.

### 2-6. 조용한 no-op

`confirm90`/`updateStage`/`setDeliverableStatus`는 `WHERE userId AND id`로 UPDATE 후 **매칭 여부와 무관하게 `{ success: true }`를 반환**한다(`routers.ts:162,173,189`). 남의 것/없는 것을 건드려도 성공으로 보인다. 영향받은 행 수를 확인하고 0이면 `NOT_FOUND`를 던지는 편이 안전하다.

## 3. 보안 · 프라이버시

- **PII를 localStorage/sessionStorage에 평문 저장**: `useAuth`(`client/src/_core/hooks/useAuth.ts`)가 `manus-runtime-user-info`에 이름·이메일 전체를 JSON으로 저장하고, `manus-cookie` 토큰을 sessionStorage에 미러링한다. XSS 시 그대로 노출된다. 프레임워크 코어(`_core`)라 수정 지양 영역이지만, 제품화 시 위협 모델에 반드시 포함해야 한다.
- `useAuth`의 `localStorage.setItem`이 **`useMemo` 안에서 실행**된다 — useMemo는 순수해야 하며 부작용을 넣으면 안 된다(React가 재실행/스킵 가능). `useEffect`로 옮겨야 한다.
- 클라이언트 라우트 가드가 화면 컴포넌트가 아니라 `DashboardLayout`에만 있다. 새 페이지를 레이아웃 없이 추가하면 가드가 빠질 수 있다. 실제 보안은 서버(`protectedProcedure`)가 지키므로 치명적이진 않으나, 라우트 레벨 가드로 일원화하는 게 낫다.

## 4. 성능 · 확장성

- 2-3(인덱스 없음)이 확장성의 최대 병목.
- `rocket.overview`(`routers.ts:50`)는 사용자의 **모든** 로켓·산출물·하위과업·세션을 한 번에 불러와 JS에서 집계한다. 한 화면(오늘/타이머/리포트) 진입마다 이 무거운 쿼리가 돈다. 지금은 4쿼리로 N+1은 피했지만, 로켓이 수백 개가 되면 페이지네이션/서버 집계가 필요하다.
- 리포트의 사용률·정확도 같은 집계는 SQL에서 계산해 요약만 내려주는 편이 확장성이 좋다.
- 타이머는 `setInterval(1000ms)` 기반이라 탭이 백그라운드로 가면 브라우저 쓰로틀링으로 시간이 밀린다. `Date.now()` 기준 경과시간 계산이 정확하다(1-2 리팩터링과 함께).

## 5. 제품화에 반드시 필요한, 지금 아예 없는 것들

1. **핵심 상태 전이 UI**(1-1) — 최우선.
2. **로켓 상세 화면** — 산출물/하위과업 목록, 체크, 단계 이동, 세션 연결이 모일 곳이 없다. 지금은 카드만 있고 클릭해도 갈 곳이 없다.
3. **에러·빈 상태·낙관적 업데이트** — mutation 실패 시 토스트는 있으나 재시도/롤백 UX가 없다.
4. **오프라인/저장 실패 복구** — 타이머 세션 기록이 네트워크 실패로 날아가면 복구 수단이 없다(1-2와 겹쳐 데이터 유실).
5. **테스트 커버리지** — 순수 함수(`shared/rocket.ts`)만 테스트되고(메트릭/날짜), 통합 테스트는 DB 없으면 스킵된다(`rocket.create.test.ts:53`). 타이머 로직·라우터 권한·소유권 검증 테스트가 없다. 1-2 같은 버그가 테스트로 안 잡힌 이유.
6. **관측성** — 서버 에러 로깅이 `console.*`뿐(`server/db.ts`). 프로덕션 구조적 로깅/모니터링 없음.
7. **접근성/국제화** — 타이머 카운트다운의 스크린리더 알림(aria-live) 없음. 날짜 표기가 `toLocaleString()` 브라우저 로캘 의존.

## 6. 코드 품질 · 유지보수

- **미사용 템플릿 잔재**: `client/src/components/AIChatBox.tsx`, `Map.tsx`, `ManusDialog.tsx`, `server/storage.ts`가 제품 코드에서 전혀 참조되지 않는다. 이로 인해 `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`, `@types/google.maps`, `axios` 같은 무거운 의존성이 딸려 있다. AGENTS.md는 "axios 말고 tRPC"라고 명시하는데 axios가 dependency에 남아 있다. 삭제 권장.
- **타입 우회**: `Today.tsx:132`, `Report.tsx:55`에서 `as unknown as RocketOverview`로 tRPC의 추론 타입을 버리고 수동 타입으로 캐스팅한다. `AppRouter`의 출력 타입(`inferRouterOutputs`)을 쓰면 서버-클라이언트 타입이 자동 동기화되고 이런 캐스팅이 사라진다. 지금은 서버 응답이 바뀌어도 클라이언트가 컴파일 에러 없이 깨진다.
- **날짜 시간대**: `todayStr()`(`shared/rocket.ts:205`)는 실행 환경 로컬 TZ 기준이다. 분류(`classifyTodayBucket`)는 클라이언트에서 도는데, 서버/클라 TZ가 다르면 "오늘" 경계가 어긋난다. 저장은 문자열 날짜라 그나마 안전하지만, 사용자 TZ를 명시적으로 정하는 게 좋다.
- **customMinutes 상한 미적용**: `Timer.tsx:193`에서 입력값을 `Number()`로만 받고 `plannedMinutes`는 `Math.max(1, ...)`로 하한만 둔다. `<input max=180>`은 브라우저 힌트일 뿐 값 제한이 아니다.
- `db.ts`에 `listProjects`, `createDeliverables`(복수형), `getRocketTask`, `listDeliverablesByTask`(단수형) 등 **호출되지 않는 헬퍼**가 여럿 있다. 죽은 코드 정리 필요.

## 7. 잘 되어 있는 점 (균형)

- `shared/rocket.ts`의 도메인 로직 분리(메트릭·날짜·라벨)와 단위 테스트는 깔끔하다. 경계값(예상 0, 실제 0) 처리도 신경 썼다.
- tRPC + Drizzle + zod 입력 검증으로 타입 안전한 뼈대를 잘 잡았고, 모든 읽기/수정 쿼리가 `userId`로 스코프되어 **횡단 접근(다른 사용자 데이터 조회)은 막혀 있다.**
- `overview`가 `Promise.all`로 4쿼리를 병렬화해 N+1을 피한 점은 좋다.
- 백색소음/알람을 Web Audio로 합성(`lib/noise.ts`)하고 AudioContext resume을 처리한 점, 모바일 스와이프·사이드바 리사이즈 등 UX 디테일은 신경 쓴 흔적이 보인다.
- 디자인 톤(그린 계열, serif 헤딩, 여백)은 일관적이고 완성도가 높다.
